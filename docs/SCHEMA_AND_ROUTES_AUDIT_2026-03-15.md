# تدقيق المسارات والجداول والسياسات — 2026-03-15

## 1) مشاكل المسارات (Routing Contract)

### A. عدم تطابق توثيق المسارات
- كان `docs/API_V1_ENDPOINTS.md` يصف مسارات كثيرة غير موجودة فعليًا في `api/v1.js`.
- تم تحديث التوثيق ليعكس فقط المسارات المنفذة داخل `api/v1.js`.

### B. وجود راوترين بمسارات `/api/v1/*`
- `api/v1.js` ينفذ عقد API مختصر.
- `lib/api-handlers.js` ينفذ عقدًا أوسع مختلفًا.
- هذا يسبب التباسًا عاليًا إذا لم يكن واضحًا أي راوتر مرتبط فعليًا في Vercel rewrites.

## 2) مشاكل الجداول (Schema Drift)

### A. جداول طوابير متضاربة بالاسم
- يوجد تعريفات/استخدامات لكل من:
  - `queue`
  - `queues`
  - `unified_queue`
- هذا يشير إلى drift تاريخي ويزيد مخاطر الأخطاء عند الترحيل أو القراءة من جدول خاطئ.

### B. جداول متقاربة الدلالة عبر migrations متعددة
- ظهرت أنماط متكررة للجداول/الوظائف عبر ملفات ترحيل متعددة بتاريخ مختلف.
- يلزم اعتماد جدول رسمي واحد لكل نطاق (خاصة queue domain) ثم ترحيل موحد.

## 3) مشاكل السياسات (RLS Policies)

### A. سياسات قديمة واسعة جدًا
- بعض المايجريشنز القديمة تفتح سياسات عامة (`USING (true)`) على جداول حساسة.
- هذا قد يكون مقبولًا تاريخيًا لكنه غير مناسب لنموذج أمني إنتاجي حديث.

### B. تكرار تعريفات سياسات عبر ملفات متعددة
- مع كثرة `DROP POLICY IF EXISTS` و `CREATE POLICY` عبر الزمن، يلزم توحيد policy source of truth.

   - دعم `exp` سواء بالثواني أو المللي ثانية لمنع رفض خاطئ للتوكنات الصحيحة.
3. تحديث `docs/API_V1_ENDPOINTS.md` ليطابق التنفيذ الحقيقي في `api/v1.js`.
